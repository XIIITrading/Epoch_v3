## Standard Feature Development Process

**IMPORTANT: Follow this exact sequence for every new feature, module, or significant change. Do not skip steps.**

### Phase 1: Pre-Build Setup

```
BEFORE WRITING ANY CODE:
1. Read this file (CLAUDE.md at root)
2. Read _architecture/SYSTEM_MAP.md — understand where the new feature fits
3. Read _architecture/DATA_FLOW.md — identify which tables you'll read/write
4. Read the target module's CLAUDE.md — understand local conventions
5. Read the specification document provided by Silva
6. If the spec references upstream or downstream modules, read their CLAUDE.md files too
```

### Phase 2: Specification Validation

Before implementation, generate a brief confirmation to Silva:

```
FILE: PLAN.md (in the working module directory)

Contents:
- SPEC REFERENCE: Which document/action page this implements
- MODULE LOCATION: Where the code will live
- INPUTS: What data this feature reads (tables, files, APIs) and from where
- TRANSFORM: What the code does to that data, in plain English
- OUTPUTS: What data this feature produces and where it goes
- DOWNSTREAM IMPACT: What other modules consume these outputs
- V2 COMPLIANCE: Confirm using shared Supabase client from 00_shared/data/supabase/
- ASSUMPTIONS: Anything not explicitly covered in the spec that you're deciding
- OPEN QUESTIONS: Anything you need Silva to clarify before proceeding
```

**STOP HERE and wait for Silva's approval before proceeding to Phase 3.**

### Phase 3: Implementation

```
RULES:
- Follow V2 patterns: import from shared.config, use shared Supabase client
- Do NOT use raw psycopg2 for new modules (legacy V1 debt — do not propagate)
- Do NOT duplicate credentials in module config files
- Match existing naming conventions in the target module
- Write incremental processors: query unprocessed records via LEFT JOIN exclusion,
  calculate, upsert with ON CONFLICT handling
- All database connections use SSL
- New Streamlit pages follow existing port conventions
- New PyQt6 tools follow existing GUI+CLI dual-mode pattern
```

### Phase 4: Testing

```
RULES:
- Run the code via terminal (non-visual testing)
- Verify record counts match expected data volume
- Verify outputs match spec description
- If the feature produces statistical results, verify:
  - Sample sizes are reported
  - Baseline comparisons are included
  - No look-ahead bias in data selection
```

### Phase 5: Validation Report

After implementation and testing, generate TWO documents:

```
FILE 1: DECISIONS.md (for Silva — plain English, no code)

Structure:
## What Was Built
- One paragraph per component describing what it does in trading/methodology terms
- No Python syntax, no technical jargon
- Use terms from the Method glossary (R-multiples, MFE/MAE, confluence, etc.)

## How It Connects
- What upstream data it reads
- What downstream modules consume its output
- Any new Supabase tables created (with plain-English column descriptions)

## Test Results
- What was tested and what passed/failed
- Record counts processed
- Any unexpected findings in the data

## Deviations from Spec
- Anything that differs from the original specification and why
- Any edge cases encountered that the spec didn't cover

## Open Questions for Silva
- Decisions that need human judgment
- Ambiguities in the spec that were resolved with assumptions (state the assumption)
- Anything that looked surprising in the data

## Architecture Document Updates Required
- List which _architecture/ files need updating based on this change
```

```
FILE 2: CLAUDE_UPDATE.md (for AI context — technical)

Structure:
## Module Context Update
- New files created (path, line count, purpose)
- New Supabase tables/columns
- New dependencies added
- New configuration parameters

## Integration Points
- Functions/classes exposed for downstream use
- Data contracts (table schemas, expected formats)
- API endpoints or CLI commands added

## Known Limitations
- What this implementation does NOT handle
- Performance constraints
- Technical debt created (if any, with justification)
```

**STOP HERE. Silva reviews DECISIONS.md and approves or requests changes.**

### Phase 6: Documentation Updates

After Silva approves, update architecture documents:

```
1. Update _architecture/SYSTEM_MAP.md if new module connections were created
2. Update _architecture/DATA_FLOW.md if new tables or data paths were added
3. Update _architecture/PIPELINE.md if the operational flow changed
4. Add an entry to _architecture/DECISIONS_LOG.md with:
   - Date
   - What was changed
   - Why (link to spec/action page)
   - What alternatives were considered
5. Update the target module's CLAUDE.md with new context
6. If the feature affects downstream modules, update their CLAUDE.md files too
```

### Phase 7: Completion Confirmation

```
Final output to Silva:
"Implementation complete. DECISIONS.md ready for review.
Architecture documents updated: [list which ones].
Ready for production approval."
```

## V2 Architecture Standards

### Database Access
```
CORRECT (V2):
from shared.data.supabase import get_client
client = get_client()

INCORRECT (V1 — do not use in new code):
import psycopg2
conn = psycopg2.connect(host=..., dbname=..., password=...)
```

### Configuration
```
CORRECT (V2):
from shared.config import credentials, settings

INCORRECT (V1 — do not use in new code):
# Duplicating credentials in module-level config.py
DB_HOST = "db.pdbmcskznoaiybdiobje.supabase.co"
```

### File Versioning
```
When revising documents or specs:
- Create new versioned file: feature_spec_v2.md
- Never overwrite the original: feature_spec.md stays as history
```

### Markdown Formatting (for documentation)
```
USE:
- Pipe-syntax tables (| Col1 | Col2 |)
- ## headers
- > blockquote callouts
- ``` code blocks

DO NOT USE:
- ASCII art tables (do not render correctly)
```

### Batch Operations
```
Supabase upserts: use ON CONFLICT handling, not delete-then-insert
Polygon.io fetches: 30-day chunks for API compliance
```

## Error Handling Convention

```
- All database operations: try/except with specific error types, log to stdout
- All API calls: retry with exponential backoff (3 attempts)
- All file operations: verify path exists before read, create directory before write
- All statistical tests: check minimum sample size before computing (30 trades minimum)
- Never silently swallow errors — log them even if the operation can continue
```
